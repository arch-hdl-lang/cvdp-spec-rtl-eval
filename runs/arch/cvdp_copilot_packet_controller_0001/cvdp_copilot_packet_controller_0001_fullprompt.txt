Problem ID: cvdp_copilot_packet_controller_0001

Natural-language specification:
Design a packet controller module in `verilog` to handle data reception and transmission. The design must implement a finite state machine (FSM) to process received packets, validate their integrity, compute appropriate responses, and manage transmission. The module should emphasize precise functionality and align with the following specifications.

---

### **Specifications**
- **Module Name**: `packet_controller`

#### **1. Input/Output Interface**
- **Inputs**:
  - `clk`: System clock signal (active on the rising edge).
  - `rst`: Active-high asynchronous reset.
  - `rx_valid_i`: Indicates valid incoming data (strobe signal).
  - `rx_data_8_i [7:0]`: 8-bit data received from the RX interface.
  - `tx_done_tick_i`: Indicates the successful transmission of a byte.
- **Outputs**:
  - `tx_start_o`: High during transmission; low otherwise.
  - `tx_data_8_o [7:0]`: 8-bit data to transmit.

---

#### **2. FSM Behavior**
- **States**:
  - `S_IDLE`: Waits for 8 bytes of data to accumulate in the buffer.
  - `S_GOT_8_BYTES`: Checks the header and computes the checksum.
  - `S_RECV_CHECKSUM`: Validates the checksum. If valid, transitions to response building; otherwise, resets to `S_IDLE`.
  - `S_BUILD_RESPONSE`: Prepares the outgoing response packet.
  - `S_SEND_FIRST_BYTE`: Initiates the response transmission by sending the first byte.
  - `S_RESPONSE_READY`: Handles the remaining response transmission.
  - **Error Handling**: Returns to `S_IDLE` on invalid checksum or unexpected header.

---

#### **3. Functional Details**
- **Packet Format**:
  - **Header**: The first 16 bits of the packet must match `16'hBACD` for valid packets.
  - **Payload**: Two 16-bit integers (`num1`, `num2`), an 8-bit opcode, and an 8-bit checksum.
  - **Checksum**: The sum of all bytes (modulo 256) must equal zero for valid packets.

- **Outgoing Response**:
  - Header: Fixed as `16'hABCD`.
  - Payload: The computed result of `num1` and `num2` based on the opcode:
    - `opcode = 8'h00`: Addition (`num1 + num2`).
    - `opcode = 8'h01`: Subtraction (`num1 - num2`).
    - Other opcodes result in `0`.
  - Checksum: Calculated to ensure the outgoing packet meets checksum validation.

---

#### **4. Error Handling**
- Invalid packets (e.g., checksum mismatch, unexpected header) reset the FSM to `S_IDLE`. 

---

#### **5. Edge Case Scenarios**
1. **Invalid Headers**:
   - Headers other than `16'hBACD` are treated as invalid, causing the FSM to reset to `S_IDLE`.
2. **Checksum Failures**:
   - Packets with incorrect checksum are discarded, resetting the FSM to `S_IDLE`.
3. **Arithmetic Operations**:
   - Handles addition and subtraction but does not explicitly address signed arithmetic or overflow/underflow scenarios.
4. **Rapid Consecutive Packets**:
   - Designed to handle valid packets one at a time without specific provisions for back-to-back packet handling.

---

#### **6. Timing Constraints**
- **FSM Synchronization**:
  - State transitions and output updates are synchronized with the system clock.
- **Transmission Latency**:
  - Transmission continues until all 5 bytes of the response are sent, synchronized to `tx_done_tick_i`.

---

### **RTL Implementation Requirements**
1. **FSM Design**:
   - Implement states (`S_IDLE`, `S_GOT_8_BYTES`, etc.) with appropriate transitions.
2. **Checksum Logic**:
   - Validate incoming packets and generate outgoing packet checksums.
3. **Arithmetic Operations**:
   - Implement addition and subtraction for `num1` and `num2` based on `opcode`.
4. **Error Handling**:
   - Reset FSM to `S_IDLE` for invalid headers or checksum mismatches.
5. **Timing and Synchronization**:
   - Ensure all state transitions and signal updates occur on the rising edge of `clk`.
